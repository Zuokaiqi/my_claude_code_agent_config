# 功能执行层拆解框架

## 核心理念

把单个功能从"一句话描述"拆到"开发能直接照着做、测试能直接照着验"的程度。靠的不是经验拍脑袋,是三个强制视角的穷举加一层落地:

- 实体怎么从创建走到终态,中间牵连谁(B1)
- 五个必问,把空状态、失败、引用、并发、撤销逐个过一遍(B2)
- 数据从哪来、到哪去、上下游接不住怎么办(B3)

最后汇总成子功能优先级清单(B4),并为每条写成可观测、可复现的验收标准AC(B5)。

## 触发判断

### 应该用

- 方向已定、产品骨架已有,要把某个具体功能拆到能开发
- 用户问一个功能要考虑哪些边界、哪些异常
- 要为功能写验收标准、定义"做到什么程度算完成"
- 开发前需要把隐性依赖、状态流转、并发冲突想清楚

### 不该用

- 还在判断这个产品/功能值不值得做 → 那是 product-strategist
- 产品还没做过结构拆解,对象/页面/数据模型没梳理过 → 先用 ooux-product-design 搭骨架
- 纯bug修复、纯样式调整 → 不需要走这套,直接进实现

## 前置衔接

如果这个功能所属的产品还没做过结构拆解(核心对象、页面、数据模型都没梳理),先用 ooux-product-design 把骨架搭出来,再回到这里拆单功能。

已经有产品结构文档的,B1 从结构文档里圈出本功能涉及的对象,然后专门补充隐性实体;没有结构文档直接进来的,B1 先做一遍完整的实体识别。

---

## B1 实体生命周期分析

B1 的增量价值是阶段A(对象结构拆解)不做的两件事:挖出隐性实体、追状态流转的牵连反应。状态机本身能复用就复用,不重画。

### 1. 识别该功能涉及的所有实体(显性加隐性)

显性实体:用户能直接看到的,订单、文章、评论。走过阶段A的,从 structure 文档里圈出本功能涉及的对象即可,不重新识别。
隐性实体:用户看不见但必须存在的,配置项、操作日志、关联关系表、缓存/快照、临时锁、版本号、操作记录。阶段A不会列这些(用户感知不到),这里是识别它们的唯一机会。

至少追问自己三次"还有没有看不见但必须存在的实体"。隐性实体漏了,开发到一半才发现要补表、补锁,返工成本最高。

### 2. 生命周期状态机:复用优先,不重画

- **走过阶段A、且阶段A已为这个对象画过状态机**:直接引用,不重画。在 ac 文档里写明「状态机见 structure 文档的某对象」
- **隐性实体,或直接进阶段B没走过阶段A的对象**:这里才补画。创建 → 各种中间状态 → 终态(归档/删除),每个状态变更标注谁触发(用户操作/系统自动/管理员操作)

补画时格式(以订单为例):
```
待支付 → 已支付 → 备货中 → 配送中 → 已签收
                                      ↘ 已取消(从待支付/已支付可流转)
                                      ↘ 退款中 → 已退款
```

### 3. 在每个状态转换节点追问牵连关系(B1 的重点)

不管状态机是引用来的还是新画的,每个转换节点都要追问:

- 这个状态变化影响哪些其他实体
- 其他实体的状态变化反过来影响这个实体吗
- 这个实体被删除/归档时,引用它的其他实体怎么处理

这一步是阶段A两样都不做的:阶段A的状态拆解只看单个对象内部流转,关联关系只看静态关系,都不覆盖状态变化时的跨实体动态连锁。

输出格式:
```
实体:[名称]
生命周期:状态A → 状态B → 状态C → ...
牵连关系:
- 从状态X到状态Y时 → 影响[其他实体]的[什么行为]
- 被[其他实体]引用时 → 删除/修改需要[怎么处理]
```

## B2 五个必问

对每个功能强制回答以下五个问题,每个问题给出具体场景和处理方案,不允许回答"暂不考虑"。

**Q1 空状态**:用户第一次使用、没有任何数据/内容/历史的情况下,界面展示什么、能做什么、不能做什么。前置条件没满足时(依赖数据还没准备好),按钮/入口如何表现。

**Q2 操作失败**:至少3种失败场景(网络中断、权限不足、数据冲突、超时、依赖服务不可用),每种场景下用户看到什么、系统做什么、已经执行的部分操作要不要回滚。

**Q3 数据被引用**:这个功能创建或修改的数据还有谁在用,被其他功能/其他用户/外部系统引用时修改和删除如何处理,是否存在数据泄露到不应该看到的人的风险。

**Q4 并发**:两个用户(或同一用户在两个设备)同时操作同一份数据会怎样,需要锁机制还是合并机制,冲突时谁的操作优先。

**Q5 后悔/撤销**:用户执行后发现错误能不能撤销,撤销范围是什么(只撤销自己的操作,还是连带影响一起回退),有没有不可逆操作需要二次确认,删除是软删除(回收站)还是硬删除。

输出格式:
```
Q1 空状态:[具体场景 → 处理方案]
Q2 失败:
  - 场景1:[描述 → 处理方案]
  - 场景2:[描述 → 处理方案]
  - 场景3:[描述 → 处理方案]
Q3 被引用:[哪些地方引用 → 修改/删除时的处理策略]
Q4 并发:[冲突场景 → 解决机制]
Q5 后悔:[可撤销范围 → 不可逆操作清单 → 二次确认策略]
```

## B3 输入输出链

1. 列出所有输入来源:数据从哪来、触发条件、依赖哪些前置操作
2. 列出所有输出去向:数据被谁消费、触发什么下游动作、影响哪些其他模块
3. 对每条输入追问:上游没准备好怎么办、格式不对怎么办、被删除了怎么办
4. 对每条输出追问:下游接不住怎么办、不需要了怎么办、数据量超预期怎么办

输出格式:
```
输入:
  - [来源1] → 上游风险:[风险描述 → 应对方案]
  - [来源2] → 上游风险:[风险描述 → 应对方案]
输出:
  - [去向1] → 下游风险:[风险描述 → 应对方案]
  - [去向2] → 下游风险:[风险描述 → 应对方案]
```

## B4 子功能优先级清单

B1到B3跑完后,汇总所有发现的子功能和待决策项,按优先级分类:

```
必须在V1实现(不做会导致功能不可用或数据错误):
1. ...

应该在V1实现(不做会导致体验明显缺陷):
1. ...

可以延后(V2考虑):
1. ...

待决策(需要产品负责人判断):
1. [问题描述] → 选项A vs 选项B → 各自影响
```

如果子功能超过15个,主动建议拆为多个独立功能分批实现。

---

## B5 验收标准(AC)

为"必须V1"和"应该V1"每条写AC。这是实现的硬性目标,也是审查通过与否的判定依据。没有AC,实现会偷工,审查没法判定通过与否。

每条AC的硬性要求:

1. **可观测**:从代码、UI、日志、数据库状态中能直接看出是否满足,禁用"合理""友好""正常""优化"这类无法判定的词
2. **可复现**:写明前置条件加操作步骤加期望结果,第三方按步骤能复现验证
3. **覆盖三类**:
   - 主流程正例:什么输入→什么输出/状态变化
   - 边界与异常:B2识别的空状态、失败、引用、并发、撤销每条都要有对应AC
   - 数据/接口契约:涉及接口、数据库、文件格式时明确字段名、类型、必填/可空、约束

输出格式:
```
## 验收标准

### AC-1:<子功能名>
- 前置:<前置数据/状态>
- 操作:<具体操作步骤>
- 期望:<可观测的具体结果,含字段值、UI元素、状态码等>
- 关联子功能:<对应清单中第几条>

### AC-2:<边界场景名>
- 前置:...
- 操作:...
- 期望:...
- 关联子功能:...
```

不合格的AC(对照修正):
- 「失败时给出友好提示」→ 「网络中断时按钮变灰,下方红色文字显示『网络异常,点击重试』」
- 「正确处理并发」→ 「两个用户同时编辑同一条记录提交时,后提交者收到409状态码,提示『数据已被他人修改,请刷新』」
- 「优化空状态体验」→ 「列表为空时居中展示插图加文案『还没有内容,点击右上角创建』加创建按钮」

## 工作规则

1. 每个步骤完成后暂停等确认或补充,不一口气全输出
2. 描述太模糊时,先列出理解和假设,确认后再开始拆解
3. 不假设任何"显然"的处理方式(比如"删除当然是软删除"),明确列出让用户决策
4. 牵连关系涉及到平台中尚不存在的模块(通知系统、权限系统等)时,明确标注为"新增基础设施依赖"
5. 拆解结果必须具体可执行,不出现"需要进一步讨论"这种模糊结论,要决策就列选项和影响
6. 输出时使用表格和清单格式,方便逐条核对

## 兜底处理

- **实体是否独立存在有争议**:给两种方案的利弊对比让用户选。例:「地址」作为「用户」的属性(简单但不灵活),还是独立成对象(支持多地址管理但复杂度增加),看产品是否需要多地址管理。
- **用户想跳过某些步骤**:允许跳过但提示后果。跳过状态拆解→后面开发时容易漏异常流程;跳过五问→边界条件会留坑;跳过AC→实现没有硬性目标,容易偷工。

## 语言规范

输出会被人当作正常文字阅读,避开以下AI腔(完整规则见 `~/.claude/rules/no_ai_style.md`,需要时 Read):

- 自我标榜:「老实说」「说实话」「坦白讲」「不夸张地说」
- 解释自己用意:「我不是要X,而是要Y」这种自我评注
- 结尾反问加主动规划:「要不要我……」「想不想我帮你……」
- 禁用符号:不用破折号(——)、不用双引号("")
- 油腻煽情:「掏心窝」「肺腑之言」「走心」
- 表态铺垫:「与其瞎说不如我去查」这类宣告下一步的话
- 附和措辞:「说得对」「确实」「没错」,直接接内容或不说
- 中文与数字/英文/代码关键词之间不加空格
- 中文语境用中文标点,不混入英文逗号句号冒号
