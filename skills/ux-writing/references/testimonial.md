# 用户证言写法

用户证言的本质不是夸奖，是**证据**。读者读证言时心里在算账：这个用户跟我像吗？他描述的痛点我有吗？他说的结果可信吗？

「非常好用，强烈推荐」给的不是证据，是 spam。

## 三段式结构（before-during-after）

每条证言至少包含这三段：

1. **Before（卡在哪）**：具体的问题。不是「效率低」，是「每月花 40 小时整理产品文档，写完连自己都不想读」
2. **During（怎么用的）**：具体的使用场景或操作。不是「使用了 XX」，是「把过去半年发的 8 封邮件喂进去当样本」
3. **After（具体结果）**：具体数字或具体变化。不是「显著提升」，是「写文档时间从每月 40 小时压到 8 小时」

三段不一定按顺序，可以倒叙（先讲结果再讲怎么做到），但三段都要在。

## 必备元素

### 1. 具体数字

数字让证言瞬间可信。挑一类放进去：

| 类型 | 例 |
|------|---|
| 百分比 | 转化率从 1.8% 涨到 4.2% |
| 时间对比 | 写周报从 1 小时到 5 分钟 |
| 金额 | 每月省 3000 元外包文案费 |
| 次数 | 这两个月发了 11 篇博客 |
| 数量 | 客户邮件打开率 52%、48%、55%、51% |
| 时长 | 上线后 48 小时 |
| 规模 | 5 个工程师团队 |

至少一个数字，最好两到三个。

### 2. 真名 + 真公司 + 真角色

匿名证言可信度低。署名格式：

```
—— Mike，独立开发者，做了 3 个 SaaS
—— Lin，30 人初创团队的技术负责人
—— 张明，前字节跳动 AI 产品负责人
```

如果是早期产品没法用真名，至少写清楚：

- 角色（开发者 / 产品经理 / 运营 / 老板）
- 公司类型 / 团队规模（10 人创业团队 / 上市公司 / 独立开发者）
- 业务（做 SaaS / 写技术博客 / 做电商）

### 3. 时间锚

「用了 3 个月」比「最近一直在用」可信。「使用 2 周后」是早期用户口吻。

### 4. 不可替代的细节

只有这个用户在这个场景下会说的话。例：

- 「CEO 烦得要死」（具体到 CEO 这个角色的反应）
- 「我之前给落地页写文案要憋一整个下午」（独立开发者的真实状态）
- 「这两个月我发了 11 篇」（技术博主的更新节奏）

这种细节不可能从模板里来。如果你写的证言把名字改成竞品也成立，证言假。

## 口语化要点

证言要像真人说话，不像 PR 稿。

### 用「我」开头，多次出现「我」

- ✅「我之前给落地页写文案要憋一整个下午」
- ❌「该产品大幅提升了文案编写效率」

### 用日常用词

| 装腔 | 人话 |
|------|------|
| 显著提升 | 涨了 3 倍 |
| 极大优化 | 快多了 |
| 深度赋能 | 帮我解决了 X |
| 完美解决 | 修好了 |
| 强烈推荐 | （删，让数据说） |
| 不可或缺 | 现在每天都用 |

### 可以有小瑕疵

完美的证言反而假。真实的证言通常带一点保留、犹豫、转折。

- 「一开始觉得贵，用了一个月觉得每月省下来的时间值这个价」
- 「中文支持还在改善，但英文文档生成已经够我用」
- 「不是所有功能都用，主要用其中 X 这一块，每周省 5 小时」

### 有起伏

平铺直叙的证言读起来像稿子。带情绪、转折、对比的证言读起来像人。

- 「之前每次产品更新邮件都要 CEO 亲自动笔，他烦得要死。现在我们把他写过的邮件做成样本，工程师写完跑一遍，CEO 只花 2 分钟读就发了。」

## 反例对照

### 完全没用的证言

| 反例 | 问题 |
|------|------|
| 「非常好用，强烈推荐」 | 0 信息量 |
| 「这是我用过最好的产品」 | 没说哪里好 |
| 「极大提升了工作效率」 | 没数字、没场景 |
| 「期待更多功能」 | 是抱怨不是证言 |
| 「该产品深度赋能业务」 | AI 写的 |

### 改后的版本

| 反例 | 改 |
|------|----|
| 「非常好用」 | 「写文档从每周 6 小时到 1 小时，省下的时间做了 2 个新功能」 |
| 「极大提升效率」 | 「上线一周注册转化率从 1.8% 涨到 3.1%」 |
| 「这是我用过最好的产品」 | 「之前试过 3 个同类工具都没解决我的问题，这个 30 分钟就把我的样本学会了」 |

## 三种长度版本

### 超短版（首屏 social proof，30-50 字）

用最强的一个数字 + 短引语。

> 「写技术博客从两天压到大半天，订阅数从 800 涨到 2300」
> —— 后端架构博主

### 标准版（产品页中段，80-120 字）

完整三段式。

> 「我之前给落地页写文案要憋一整个下午，写完自己读都觉得是技术文档。用 ToneShift 之后我先把自己的 Twitter 喂进去当样本，落地页文案 20 分钟搞定，上线后 48 小时转化率从 1.8% 涨到 4.2%。最关键的是文案我自己看得下去，不是那种一眼 AI 的味道。」
> —— Mike，独立开发者，做了 3 个 SaaS

### 案例版（专门的 case study 页，300-500 字）

三段式展开，加背景、细节、数据图。一般独立成页。

## 排版规范

```
"<引语>"

—— <姓名>，<角色>，<公司或业务>
```

要点：

- 引号包裹引语（用中文引号「」或英文引号 ""，全局一致）
- 署名用全角破折号 ——（中文上下文里）
- 头像/Logo 可选，有的话放在署名前
- 不要给证言加感叹号或表情符号

## 收集真实证言的 prompt

如果你在找用户要证言，问这些问题能问出有用的内容：

1. 用之前你处理 [问题] 是怎么做的？花多少时间？
2. 第一次用我们产品是什么场景？解决了什么？
3. 用了多久？现在每周/每月用几次？
4. 如果要给同行推荐，你会怎么形容这个产品？
5. 你最近用它做过哪件具体的事？花了多少时间？得到什么结果？

避免问「你觉得好用吗？」（必然得到「好用」），改问「你最近一次用它是什么时候？」

## 自检清单

- [ ] 有具体数字吗（至少一个百分比/时间对比/金额）？
- [ ] 有真名或具体角色身份吗？
- [ ] 有不可替代的场景细节吗？
- [ ] 用「我」开头吗？
- [ ] 朗读起来像真人说话吗？
- [ ] 把产品名改成竞品，证言还成立吗？成立 = 假证言
- [ ] 有没有「赋能」「打造」「重新定义」等 AI 腔？
