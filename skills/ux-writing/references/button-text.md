# 按钮文案规则

## 核心原则

1. 动词开头：「保存修改」「删除订单」「立即下单」
2. 主谓宾相对完整：避免单字「确定」「OK」「完成」
3. 避免AI腔通用词：禁用「提交/确认/完成/OK/好的」单独使用
4. 主操作和次操作语义区分：「立即购买」（主）vs「加入购物车」（次）

## 反例对照

| 反例 | 改进 | 原因 |
|------|------|------|
| 提交 | 保存修改 / 立即下单 / 发布文章 | 「提交」是泛用词，不告诉用户做了什么 |
| 确认 | 确认删除 / 确认取消订单 | 单独的「确认」不清楚确认什么 |
| OK / 好的 | 知道了 / 关闭 / 立即查看 | OK 是占位文案，没有信息 |
| 完成 | 完成创建 / 结束流程 | 单独的「完成」语义弱 |
| 操作 | 编辑 / 删除 / 复制 | 表头里写「操作」是开发者偷懒 |
| 立即体验 | 开始记账 / 创建第一篇 | AI腔默认词，没有具体动作 |
| 立即开启 | 开启自动备份 / 启用通知 | 开启什么要写清楚 |
| 一键生成 | 生成周报 / 生成PPT | 「一键」是营销词不是动作 |
| 马上试试 | 试试AI总结 / 试试自动分类 | 试试什么要写清楚 |

## 边界情况

- 弹窗的「取消」按钮可以保留（用户惯例强）
- 「下一步」「上一步」在多步流程里清晰可保留
- 业务术语在按钮里要注意用户是否懂（「开票」「核销」「冲账」可能要更通俗）
- 表格行内的「编辑/删除/复制」可以保持单动词（上下文足够清楚）

## 按钮状态文案

| 状态 | 文案规则 | 例 |
|------|---------|---|
| 加载中 | 动词进行时 + 「...」 | 保存中... / 下单中... |
| 成功后 | 按钮变「已XX」，2秒后恢复 | 已保存（2秒）→ 保存修改 |
| 禁用态 | 配合 tooltip 说明为什么禁用 | 按钮变灰 + tooltip「填完必填项才能保存」 |
| 危险操作 | 红色填充 + 后果明确 | 永久删除 / 立即清空 |

## 主次按钮区分

- 主按钮：填充色高对比，文案是核心动作
- 次按钮：描边或文字，文案是辅助动作
- 危险按钮：红色或警示色，文案点出后果
- 同一组里不要出现两个填充色主按钮

## 项目一致性

- 写新按钮前先 Grep 项目里同类按钮（如表单保存类、删除类、确认类）
- 沿用项目既有措辞和语气，不引入新风格
- 项目用「保存」就不在新地方用「提交」
