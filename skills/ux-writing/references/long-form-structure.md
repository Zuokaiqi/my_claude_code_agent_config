# 长文案结构与节奏

长文案（落地页、课程详情页、产品介绍页、博客文章）≠ 把短文案堆起来。需要专门的结构和节奏控制，否则容易写出 AI 腔的"漂亮但没人读"的稿子。

## 信息架构

### 倒金字塔：结论 → 理由 → 细节

读者只读前 3 句就走的概率很高。把最值钱的信息放最前面。

- 第 1 句：这是什么、给谁的
- 第 2-3 句：跟其他选择的差异
- 第 4 句开始：展开细节、场景、证据

反例（AI 腔典型）：「随着 AI 技术的快速发展，越来越多的工作流可以被自动化重塑。本课程旨在帮助学员系统掌握……」—— 前 30 字没传递任何信息。

### F-pattern：读者按 F 形扫读

读者不是逐字读，是按 F 形扫读（顶部横扫一段，左侧纵向跳读）。所以：

- 每段第一句独立成立（不依赖上一段的语境）
- 关键词出现在每段头部
- 列表/小标题/加粗放重要信息
- 不要重要信息埋在长段落中间

### 一段一个主旨

每段表达一件事，不超过 5 句话。如果一段有两件事，拆成两段。

## 句子节奏

### 短句长句交错

整段全是 25 字以上的长句 = AI 腔。整段全是 8 字以内短句 = 朋友圈文案腔。

规律节奏：长-短-长-短-中。或：短-短-长-短。打破任何规律的连续 3 句相同句长。

例（节奏好）：
> 周六晚上，你把 SaaS 的落地页骨架搭完了。功能列表你列得出来，但写成卖点就卡住。每句都像 API 文档。打开 ToneShift，把功能列表贴进去，选「面向独立开发者，平实不夸张」，三分钟拿到一版改写。

短（周六晚上 9 字）→ 中（25 字）→ 短（8 字）→ 长（40+ 字）。读起来像人在讲话。

### 避免连续 3 个相同句式

反例：「这门课教你 X。这门课让你 Y。这门课帮你 Z。」三个「这门课」是 AI 模板。

改：「这门课教你 X。学完能 Y。结课时手里会有 Z。」

## 用具体代替抽象（5 种手法）

抽象描述是长文案最大的 AI 腔来源。所有形容词、副词、宣称，都要替换成具体的东西。

### 手法 1：数字代替形容词

| 抽象 | 具体 |
|------|------|
| 高效 | 3 秒返回结果 |
| 提升团队效率 | 把每周 20 小时压到 3 小时 |
| 显著提高转化率 | 转化率从 1.8% 涨到 4.2% |
| 节省大量时间 | 写周报从 1 小时到 5 分钟 |
| 性能优秀 | 5000 页文档秒开 |

### 手法 2：例子代替概念

| 抽象 | 具体 |
|------|------|
| 灵活配置 | 周一发周报，周三发月报，可以分别配 |
| 多场景适用 | 写 PRD、改老代码、做调研、出周报都能用 |
| 智能改写 | 输入「This tool helps you parse CSV」→ 输出「CSV 丢进来，三秒拿到字段」 |
| 智能分类 | 把 200 条用户反馈自动分成 bug、需求、好评三类 |

### 手法 3：场景代替宣称

| 抽象 | 具体 |
|------|------|
| 提升用户体验 | 加载从 3 秒到 0.5 秒，再也不用看菊花 |
| 增强协作 | PM 改完 PRD，工程师在 Slack 直接收到 diff |
| 加强安全 | 数据存在你自己服务器，我们看不到 |

### 手法 4：类比代替术语

| 术语 | 类比 |
|------|------|
| Vector embedding | 把文字变成一串坐标，距离近的意思就相近 |
| RAG | 先去你的资料库里找相关段落，再让大模型基于这些段落回答 |
| Function calling | 让大模型自己决定该调哪个 API |
| Tree-shaking | 没被用到的代码不打进包 |

类比要选目标读者真正熟悉的概念，不要用更冷门的术语解释术语。

### 手法 5：引语代替描述

不写「用户反馈很好」，写用户怎么说：

> 「我之前给落地页写文案要憋一整个下午，写完自己读都觉得是技术文档。用 ToneShift 后 20 分钟搞定，上线后 48 小时转化率从 1.8% 涨到 4.2%。」
>
> —— Mike，独立开发者，做了 3 个 SaaS

## AI 腔结构反例

### 反「不是 X，是 Y」对比

这个句式本身可以用，但被否定的那个 X 必须是读者真实会有的认知。

- ❌「我们不是工具，是伙伴」—— 没人会把工具理解成伙伴
- ❌「不是核对错别字，是核对结论」—— 没人会把审稿理解成对错别字
- ✅「不是 Notion 的替代品，是 Notion 用户的快速版」—— 「替代品」是真实存在的误解

### 反 X-theater / actually-X 套话

用句式制造深刻感、实则没信息的 instant slop，一律禁：

- 英文：「productivity theater」「engagement theater」（XX 剧场）、「not just X, it's Y」、「actually X」当卖点
- 中文：「真正意义上的 X」「不只是 X，更是 Y」「重新想象 X」「X 不止于 X」

删掉，直接说具体做了什么。

### 反 meta-commentary

不要在文案里描述文案本身。

- ❌「在这个章节里，我们将探讨……」
- ❌「让我们一起来看看……」
- ❌「综上所述」「总而言之」「最后」（总结词）
- ✅ 直接开始讲

### 反 formal transitions 滥用

「此外」「另外」「同时」「然而」「值得注意的是」过度连接，是 AI 翻译腔典型。

- 一段一个意思，段与段之间不需要每次都用连接词
- 必要时用「但」「不过」「还有」这类口语词代替

### 反翻译腔结构

| 翻译腔 | 改 |
|--------|----|
| 让我们一起 | （删） |
| 你将拥有 | 你能 |
| 我们将 | 我们要 / 接下来 |
| 您的不二选择 | （删整句） |
| 在 XX 中 | XX 里 |
| 通过 XX 来 XX | 用 XX |
| 对于 XX 而言 | XX |

### 反总结式收尾

不要在长文案末尾写「相信通过这次的学习」「这就是 XX 的全部」「希望对你有帮助」。

长文案的结尾通常是行动指引（CTA）或证据（证言、数据），不是总结。

## 翻译腔自检

朗读一遍长文案，问自己：

- 这段我会跟朋友这样说吗？不会就是翻译腔
- 主语是「我们」「您」「让我们」开头吗？多于 3 处就过量
- 有没有「将」「应当」「值得」滥用？这些是翻译腔信号词

## 段落节奏自检

写完一段：

- 句子长度有起伏吗（最长最短差 3 倍以上）？
- 第一句独立成立吗（不依赖上段语境）？
- 有具体的人/物/数字吗？没有就拆改
- 朗读 3 遍还顺畅吗？拗口就改

## 整篇自检（写完后过一遍）

1. **前 3 句砍掉，文章还成立吗**：成立说明前 3 句是废话铺垫
2. **每段开头能扫读吗**：F-pattern 扫一眼能抓住每段意思
3. **黑名单词清单全过一遍**：Grep 检查「赋能/打造/一站式/沉浸式/解锁」等
4. **抄到竞品页面上不成立**：抄过去也通顺就是没差异化
5. **有几个数字几个例子几个引语**：少于 3 个具体物 = 过抽象
6. **去掉所有形容词后，意思还完整吗**：完整说明形容词没必要存在
