# 错误信息文案规则

## 核心原则

1. 不责怪用户，描述系统状态：「我们没识别到这个格式」而非「您输入的内容有误」
2. 说清楚错在哪：哪个字段、什么问题
3. 给出下一步：用户可以做什么来解决
4. 不暴露技术细节：避免「errorCode 5023」「null pointer exception」「请求失败 500」直接抛给用户

## 反例对照

| 反例 | 改进 |
|------|------|
| 操作失败 | 网络似乎断了，刷新试试 |
| 系统繁忙，请稍后重试 | 服务暂时无法响应，30秒后再试 |
| 您输入的内容有误 | 手机号需要11位数字，当前少了 |
| 请检查您的输入 | 邮箱格式不对，需要含@和域名 |
| Server Error 500 | 服务出问题了，我们已收到告警，请稍后重试 |
| 您没有权限 | 这个操作需要管理员账号，找xxx申请 |
| 请求参数错误 | 缺少必填的「订单号」字段 |
| 网络异常 | 网络似乎断了，检查Wi-Fi后下拉刷新 |
| 用户名已存在 | 这个用户名已被注册，换一个或直接登录 |
| 密码错误 | 密码不对，再试一次或点这里找回 |

## 错误信息出现位置

| 错误类型 | 显示位置 | 持续时间 |
|---------|---------|---------|
| 表单字段错误 | 出错字段下方就近显示 | 持续直到用户改正 |
| 全局服务错误 | 顶部 banner 或 toast | 直到用户操作或网络恢复 |
| 不可恢复错误 | modal + 解决路径 | 直到用户主动关闭 |
| 网络断开 | 顶部 banner | 直到网络恢复自动消失 |

避免：错误信息在顶部，但用户在底部填表（找不到出错字段）；错误toast 1秒消失（用户没看清）。

## 错误归因

| 谁的错 | 文案语气 | 例 |
|--------|---------|---|
| 用户输入问题 | 解释为主，说明正确格式 | 手机号需要11位数字 |
| 用户权限不足 | 客观说明，给申请路径 | 这个操作需要管理员权限，找xxx申请 |
| 系统问题（网络、服务、bug） | 承认是系统问题，给重试或反馈入口 | 服务暂时不可用，我们在修复 |
| 不确定 | 默认按系统问题处理 | 出了点问题，刷新试试 |

不让用户自责。哪怕真是用户输错了，也用「我们没识别到」「这个格式我们暂不支持」这种语气。

## 错误信息的下一步

每条错误信息要带「用户可以做什么」：

| 错误 | 没下一步（反例） | 给下一步（正例） |
|------|----------------|----------------|
| 网络断 | 网络异常 | 网络似乎断了，下拉刷新 |
| 服务挂 | 服务不可用 | 服务暂时不可用，点这里反馈 |
| 权限不足 | 您没有权限 | 这个操作需要登录，去登录 |
| 数据冲突 | 操作冲突 | 别人刚改过这条数据，刷新看最新版 |
| 超时 | 请求超时 | 网络慢，再试一次或换网络 |

## 不要做的事

- 不在错误信息里抛技术栈细节（API path、status code、stack trace）
- 不用「请」字滥用：「请稍后重试」可以删掉「请」
- 不连发感叹号：「失败了！！」是反例
- 不用全英文报错抛给中文用户
- 错误信息不写成「您的请求未能成功完成」这种翻译腔
