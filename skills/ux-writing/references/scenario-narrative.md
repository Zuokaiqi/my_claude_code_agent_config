# 场景叙述与功能描述

落地页里最容易写得抽象的两部分：场景叙述（「真实使用场景」「用户故事」）和功能描述（「核心功能」「能力介绍」）。这两类都需要落到具体的人和具体的输入输出，不能停在「智能化」「一键」「全方位」。

## 场景叙述：卡在哪 → 怎么解决 → 产出什么

每个场景必须有这三段。少一段就是空话。

### 卡在哪（场景细节五要素）

具体到读者能在场景里认出自己。五要素覆盖到 3-4 个就够：

| 要素 | 例 |
|------|---|
| 时间 | 周六晚上 9 点 / 周二早 10 点 / 上线前 2 小时 / 客户电话挂掉后 5 分钟 |
| 地点 | 地铁里 / 办公室 / 床上 / Slack 里 / 开会中途 |
| 设备/工具 | 手机 / VSCode / Figma / Notion |
| 情境 | PM 突然找你 / CEO 在群里 @ 你 / 客户投诉过来 / 你刚写完 demo |
| 心情 | 焦虑 / 目标导向 / 不耐烦 / 困 |

例：

> 周二下午 4 点，CEO 在 Slack @ 你：明早 9 点要给 2000 个早期用户发产品更新邮件，今晚之前给个版本。你看着这周的 8 个改动，不知道哪个该突出讲，哪个一句话带过。

这段四要素齐全（时间、地点/工具、情境、心情），读者能立刻代入。

### 怎么解决（具体操作）

不写「使用了 XX 功能」，写具体的操作步骤和选择：

> 把这周改动列成 8 条 bullet，打开 ToneShift，选「面向早期用户，承认问题，说清改了什么」，3 分钟拿到一版 400 字的邮件初稿。

具体到：

- 输入是什么（8 条 bullet）
- 在产品里做了什么操作（选 XX 模式）
- 花了多少时间（3 分钟）
- 拿到什么形态的输出（400 字邮件初稿）

### 产出什么（可量化或可感知）

不写「效果显著」，写具体结果：

> 改了两处措辞发给 CEO，CEO 没再改就发出去了。

或量化结果：

> 邮件发出去 48 小时，2000 个用户里 1042 个打开，去年同期同类邮件打开率 25%。

或感知层面的结果：

> 你周三早上 9 点之前发完邮件，下班前没再被催。

## 场景叙述完整范例

```
## 场景：周四下午被 CEO 突袭

周四下午 4 点，CEO 突然让他给 2000 个早期用户发邮件，"今晚发出去"。
他从来没写过用户邮件，打开文档憋了半小时只憋出："本周我们上线了批量
导出功能，欢迎使用。"

他把 CEO 过去半年写的 8 封邮件丢给 ToneShift 当样本，把这周 3 个更
新点列成 bullet 丢进去。出来的版本开头是 CEO 常用的"上周有用户问我
们……"，中间把 3 个更新讲成 3 个故事，结尾留了 CEO 一贯的彩蛋句。

他改了 2 个产品名就发了。CEO 在 Slack 回了个 👍。下周三 CEO 又让他
写邮件，他自己上手就行了。
```

三段都有具体细节：时间、人物状态、操作、结果。结尾的「下周三 CEO 又让他写邮件」是延展，比直接结束更有说服力。

## 功能描述：输入 → 输出对照

功能描述最大的陷阱是用「智能」「一键」「自动」概括完事。这些词不传递信息。

### 标准结构

每个功能至少给一组「输入 → 输出」的具体例子。

| 反例（抽象） | 正例（具体输入输出） |
|------------|-------------------|
| 智能改写 | 输入「This tool helps you parse CSV files fast」→ 输出「CSV 文件丢进来，三秒拿到你要的字段」 |
| 一键润色 | 输入开发者腔的产品描述（「本工具基于 LLM 实现自动文案改写」）→ 输出（「不用每段文案都自己琢磨语气，挑一个风格，它替你换措辞」） |
| 智能分类 | 输入 200 条用户反馈 → 自动分成「Bug 32 条 / 需求 87 条 / 好评 81 条」三组 |
| 全自动归档 | 设置规则「凡是发件人含 @client.com 的邮件归到「客户」文件夹」→ 之后所有匹配邮件自动归类 |
| AI 摘要 | 输入 6000 字技术博客 → 输出 200 字摘要，保留 3 个代码示例和 1 个性能数字 |

### 输入输出三要素

每组例子至少展示：

1. **真实的输入**（用真实例子，不用 placeholder「您的内容」）
2. **明确的产出**（具体的字数、形态、字段）
3. **关键约束**（保留什么、改变什么）

例：

> **段落级控制**
>
> 你贴入一篇 1000 字的草稿，对其中 3 段不满意，只选中那几段重写，其余保留。改写时可以指定单段约束（这段要短、这段保留代码块、这段加一个生活化比喻），不必整篇推倒。

要素齐全：输入（1000 字草稿）+ 操作（选 3 段）+ 输出（其余保留）+ 约束（每段单独指定）。

## 功能名写法

功能名最容易 AI 化。准则：

- **具体动作 + 对象**，不用形容词
- 10 字以内
- 不要「智能」「自动」「一键」开头
- 用读者能一眼看懂的词，不发明新词

| 反例 | 正例 |
|------|------|
| 智能改写 | 调性改写 / 草稿润色 |
| 一键优化 | 段落级控制 / 单段重写 |
| 全自动分类 | 反馈分类 / 邮件归档 |
| AI 助手 | （太泛，不要） |
| 沉浸式编辑器 | 全屏写作 |
| 智能推荐 | 相似文章推荐 |

## 第二人称 vs 第三人称

场景叙述选第二人称（"你"）还是第三人称（"他/她"），二选一并全局保持一致。

### 第二人称（"你"）

读者代入感强，适合：

- 产品落地页里的「使用场景」
- 课程详情页里的「适合谁」
- 教程类内容

例：「周六晚上 9 点，你把 SaaS 的落地页骨架搭完了……」

### 第三人称（"他/她" 或具体名字）

更像故事/案例，适合：

- 案例研究（case study）
- 长篇用户故事
- B 端销售页

例：「Mike 是个独立开发者，做了 3 个 SaaS……」

混用是 AI 腔典型征兆：一段「你」一段「他」，读者出戏。

## 反例：没有场景的功能描述

下面这种写法是典型的 AI 占位文案，没有任何具体场景，读完不知道这个功能给谁用、什么时候用、解决什么问题：

```
## 智能创作助手

通过先进的 AI 技术，为您打造一站式内容创作体验。无论是日常工作还
是创意写作，都能轻松应对，全方位提升您的创作效率。
```

诊断：

- 「智能」「先进的 AI 技术」「打造」「一站式」「全方位」全是 AI 腔
- 没有具体输入输出
- 没有具体场景
- 没有具体读者
- 「无论是 X 还是 Y」万能句式

改：

```
## 把粗稿改成成品

你写代码顺手，写文案别扭。把功能要点列成 bullet 丢进来，选一种调
性（"克制专业" / "热血兴奋" / "技术博客"），10 分钟拿到一版能直
接发的文案。原意和功能点不变，只改语气和措辞。
```

## 自检清单

### 场景叙述

- [ ] 有具体时间吗（周二下午 / 周六晚上）？
- [ ] 有具体地点或工具吗（Slack / VSCode / 地铁里）？
- [ ] 有具体情境吗（CEO 突然找、客户投诉）？
- [ ] 有具体卡点吗（不是「效率低」是「憋了半小时只写出一句」）？
- [ ] 解决方案有具体操作吗（选什么模式、输入什么）？
- [ ] 结果有具体数字或感知吗（不是「显著提升」）？

### 功能描述

- [ ] 每个功能有真实的输入例子吗？
- [ ] 有明确的输出例子吗？
- [ ] 功能名 10 字以内、不含「智能」「一键」「自动」？
- [ ] 朗读一遍，能听懂这个功能在干什么吗？

### 删除测试

把人物姓名、产品名、行业词删掉，这段还成立吗？还成立 = 太泛，没场景。
