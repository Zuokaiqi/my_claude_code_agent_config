# OOUX 产品对象拆解框架

## 核心理念

这套方法论的本质是**把产品设计变成一个可穷举的分析过程**，而不是靠灵感和经验。

类比数据分析中的指标拆解：
- 指标拆解：北极星指标 → 一级指标 → 二级指标 → 原子指标
- 产品拆解：**产品 → 核心对象 → 属性/状态/行为/关联 → 页面结构**

## 触发判断

### 应该触发的场景

- 用户明确说要做一个产品/app/小程序/网站，且还没进入具体编码阶段
- 用户说不知道页面该放什么、想不清楚数据结构、功能点列不全
- 用户在画原型前，想系统梳理产品结构
- 用户拿着一个模糊idea，需要变成可执行的功能清单
- 用户提到OOUX、对象拆解、产品对象分析

### 不应该触发的场景

- 用户已经有明确的PRD/原型，只是在问开发实现问题 → 那是编码任务
- 用户在讨论产品策略、商业模式、市场分析 → 那是战略层，不是结构层
- 用户只是想做一个简单的单页工具（比如计算器、转换器）→ 对象太少，直接做即可，不需要走完整框架
- 用户在迭代已有产品的某个具体功能 → 只需要针对该功能局部拆解，不需要全局走流程

### 判断用户处于哪个阶段

进入工作流程前，先判断用户当前的状态：

| 用户状态 | 信号 | 应对方式 |
|---------|------|---------|
| 只有模糊想法 | 我想做一个XX、没有具体功能描述 | 先做场景澄清（见上下文收集），再进入第一步 |
| 有初步想法 | 能说出2-3个核心功能，但没系统梳理 | 直接进入第一步提取对象 |
| 有部分梳理 | 已经列了一些功能点或画了草图 | 确认已有内容，从缺失的步骤开始补全 |
| 只卡在某一步 | 我对象列好了但不知道怎么推页面 | 跳到对应步骤，不用从头走 |

## 上下文收集

在正式开始四步拆解之前，需要先收集足够的上下文。不要在信息不足时就开始输出对象清单。

### 必须收集的信息

1. **核心场景**：用一句话描述——谁（目标用户），在什么情况下（使用场景），做什么事（核心任务），得到什么结果（价值）。如果用户说不清楚，用追问引导，例如：你这个产品主要是给谁用的？他们现在遇到什么问题？
2. **产品形态**：app / 小程序 / Web端 / 多端？这影响页面结构和交互模式的选择
3. **用户角色**：只有一种用户，还是有多种角色（比如买家和卖家、学生和老师）？多角色意味着同一个对象可能有不同的行为和视图

### 建议收集但非必须的信息

4. **竞品参考**：有没有类似的产品可以参考？竞品是最快的对象穷举来源
5. **MVP范围**：第一版要做到什么程度？是验证想法还是要上线运营？这决定对象和行为的取舍
6. **技术约束**：是否有已选定的技术栈或后端服务？这影响数据模型的建议方式

### 收集原则

- 如果用户已经在对话中提供了部分信息，可以直接复用
- 如果有不确定的信息，不要吝啬你的提问，必须信息收集完才可以开始

## 工作流程

按以下四步顺序执行。每一步都要输出结构化结果，供用户确认后再进入下一步。

---

### 第一步：提取核心对象

目标：找出用户的产品中，用户会与之交互的所有实体。

**怎么找对象：**

1. 让用户用一两句话描述产品核心场景（谁，在什么情况下，做什么事）
2. 从描述中提取所有**名词**，这些名词就是候选对象
3. 用以下过滤标准筛选：
   - 它是否有自己独立的属性？（有 → 保留）
   - 用户是否需要对它进行增删改查？（有 → 保留）
   - 它是否只是另一个对象的属性？（是 → 降级为属性，不作为独立对象）
4. 如果用户提供了竞品，可以让用户截图竞品核心页面，从页面中扫描名词来补充

**输出格式：**

```
核心对象清单：
1. [对象名] — 一句话说明是什么
2. [对象名] — 一句话说明是什么
...
```

提示用户确认：是否有遗漏或需要合并的对象。

---

### 第二步：拆解每个对象（CSMA框架）

对每个核心对象，从四个维度拆解：

#### C — Core Content（核心属性）

这个对象本身有哪些信息字段？相当于数据库建表时的字段。

思考方式：如果这个对象是一张数据库表，列名是什么？

示例（以文章为例）：标题、正文、封面图、作者、创建时间、更新时间、阅读量、点赞数

#### S — State（状态与生命周期）

这个对象从创建到结束，会经历哪些状态？状态之间怎么流转？

思考方式：画一个状态机——起点是创建，终点是删除/归档，中间有哪些节点？

示例（以订单为例）：
```
待支付 → 已支付 → 备货中 → 配送中 → 已签收
                                      ↘ 已取消（可从待支付已支付流转过来）
                                      ↘ 退款中 → 已退款
```

每个状态要标注：谁触发的状态变更（用户操作/系统自动/管理员操作）。

#### M — Metadata（元信息）

辅助管理和展示的信息，不是对象本身的内容，但影响展示和检索。

常见元信息清单（逐条过一遍，按需选取）：
- 分类/类型
- 标签
- 排序权重
- 可见性/权限
- 来源/渠道
- 版本号

#### A — Actions（用户行为）

用户能对这个对象做哪些操作？

**万能行为清单（逐条过一遍，不要靠记忆）：**

| 行为类别 | 具体操作 |
|---------|---------|
| 创建类 | 新建、导入、复制、从模板创建 |
| 查看类 | 查看详情、查看列表、预览、搜索、筛选、排序 |
| 编辑类 | 修改内容、修改状态、批量编辑、拖拽排序 |
| 删除类 | 删除、批量删除、归档、恢复 |
| 社交类 | 点赞、评论、收藏、分享、关注、举报 |
| 导出类 | 下载、导出、打印、生成链接 |
| 协作类 | 邀请、分配、转交、@提及 |

不是每个对象都需要所有操作，但过一遍这个清单可以避免遗漏。

---

### 第三步：梳理对象间的关联关系

明确对象之间的关系类型和方向：

| 关系类型 | 含义 | 示例 |
|---------|------|------|
| 一对一 | A 唯一对应 B | 用户 ↔ 个人设置 |
| 一对多 | A 拥有多个 B | 用户 → 多篇文章 |
| 多对多 | A和B互相多对多 | 文章 ↔ 标签 |
| 从属关系 | B 依附于 A 存在 | 评论从属于文章 |

**输出格式：** 建议用关系图或表格展示，标注关系类型和方向。

关联关系直接决定了：
- 页面之间的跳转逻辑（从文章详情页可以跳到作者主页）
- 数据表的外键设计
- 列表页的筛选维度（按作者筛选文章）

---

### 第四步：从对象推导页面结构

这一步是机械性的。每个核心对象通常对应以下页面模式：

| 页面类型 | 作用 | 对应的行为 |
|---------|------|----------|
| 列表页 | 展示该对象的集合 | 搜索、筛选、排序、批量操作 |
| 详情页 | 展示单个对象的完整信息 | 查看、社交互动、状态变更 |
| 创建/编辑页 | 填写或修改对象信息 | 新建、编辑 |
| 设置/管理页 | 配置对象的元信息 | 权限、分类管理 |

**推导规则：**
1. 每个用户可直接操作的核心对象 → 至少一个列表页 + 一个详情页
2. 用户需要创建/编辑的对象 → 加一个创建/编辑页（或用弹窗代替）
3. 从属对象（如评论）通常嵌入父对象的详情页，不单独成页
4. 全局功能（搜索、通知、设置）→ 独立页面或组件

**别忘了这些容易遗漏的页面/状态：**
- 空状态页（列表无数据时显示什么）
- 加载中状态
- 错误/异常状态
- 登录/注册流程
- 引导页/新手教程
- 权限不足提示

**输出格式：** 按功能模块分组的页面清单，每个页面标注包含的对象和核心交互。

---

## 最终交付物

完成以上四步后，汇总生成：

1. **对象清单表**：所有对象及其属性、状态、行为
2. **关联关系图**：对象间的连接方式
3. **页面结构清单**：从对象推导出的完整页面列表
4. **数据模型草案**：基于对象和关联的数据库表结构（字段名、类型、关联键）

以上内容足以支撑用户进入原型设计或直接使用AI工具生成代码。

## 边界处理

### 对象粒度的判断

拆解时最常见的问题是粒度不对——太粗或太细。

**太粗的信号**：一个对象有超过15个属性，或者同时承担多种完全不同的职责。这时应该拆分。例如用户既包含账号信息又包含收货地址，地址应该独立为对象。

**太细的信号**：一个对象只有1-2个属性，且完全依附于另一个对象存在，没有独立的行为。这时应该降级为父对象的属性。例如文章标题不需要独立成对象，它是文章的属性。

**判断标准**：这个东西能不能被独立地创建、查看、删除？如果可以，它是对象；如果不可以，它是属性。

### 多角色产品的处理

当产品有多种用户角色（如买家/卖家、老师/学生）时：

1. 对象清单是共享的——订单对买家和卖家来说是同一个对象
2. 但行为（Actions）要按角色分别列出——买家可以下单，卖家可以发货
3. 状态流转中要标注每个状态变更是由哪个角色触发
4. 页面推导时，每个角色可能需要独立的列表页和详情页视图

### 涉及AI能力的产品

如果用户要做的产品涉及大模型能力（比如AI对话、智能推荐、内容生成），需要额外关注：

1. AI生成的内容本身可能是一个对象（如对话记录、生成结果），要纳入对象清单
2. AI相关的行为包括：触发生成、重新生成、编辑AI结果、反馈评价
3. 状态要包含AI特有的：生成中、生成完成、生成失败
4. 数据模型中要考虑prompt和结果的存储

### 不适用此框架的场景

以下情况不建议走完整的四步流程：

- **纯内容展示型产品**（如个人博客、作品集网站）：对象通常只有1-2个，直接列属性和页面即可
- **单一流程型工具**（如格式转换器、计算器）：没有持久化对象，按操作流程设计即可
- **已有成熟系统的局部改造**：只需要对新增功能涉及的对象做局部拆解

遇到这些场景时，简要说明为什么不需要走完整流程，直接给出合适的轻量方案。

## 兜底处理

### 用户给不出清晰场景描述时

不要反复追问，换一种方式引导：
1. 问用户你自己最近遇到的、最想用一个工具来解决的烦恼是什么
2. 或者让用户说出3个最接近的竞品，从竞品反推场景
3. 如果用户确实还在很早期的探索阶段，建议先做用户调研，暂不进入结构拆解

### 对象拆不干净时

如果用户对某些对象是否独立存在有争议，给出两种方案的利弊对比，让用户选择。例如：'地址'可以作为'用户'的属性（简单但不灵活），也可以独立成对象（支持多地址管理，但增加复杂度）。你的产品需要用户管理多个地址吗？

### 用户想跳过某些步骤时

允许跳过，但提示后果：
- 跳过状态拆解 → 后面开发时容易漏掉异常流程处理
- 跳过关联关系 → 页面跳转逻辑可能不通
- 跳过行为清单过一遍 → 可能遗漏功能点，但可以迭代补充

### 对象数量过多时

如果提取出的对象超过8个：
1. 先按核心对象和辅助对象分两层
2. 建议用户先只做核心对象（3-5个）的完整拆解作为MVP
3. 辅助对象只列名称和一句话说明，留到后续迭代

### 中途需求变更时

如果用户在拆解过程中改变了产品方向或核心场景：

1. 不要接着走，回到上下文收集重新确认场景
2. 已完成的对象拆解中，可复用的部分保留，需要调整的标注出来
3. 让用户确认新的对象清单后再继续

---

## 使用注意事项

- 每一步完成后都要让用户确认再继续，不要一口气输出全部内容
- 输出时使用表格和清单格式，方便用户逐条核对
- 鼓励用户提供竞品参考，从竞品页面中扫描对象是最快的穷举方式
- 语言风格保持简洁直接，用类比解释概念（尤其对非技术背景的用户），避免堆砌术语
