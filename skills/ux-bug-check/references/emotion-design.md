# 情绪类

## 视角

用户用这份产物时的情绪体验。被责怪了吗、被吓到了吗、有没有安全感、出错后觉得是自己的错还是产品的错。

## 检查项

1. 错误信息语气是否在责怪用户（参考 ux-writing/error-messages.md 的归因原则）
2. 不可逆操作（删除、清空、退款）有二次确认吗
3. 失败后有重试入口吗
4. 删除后有撤销窗口吗
5. 操作成功反馈是否适度（过于浮夸或过于冷漠都不合适）
6. 危险操作的视觉警告是否到位（红色、警示icon、文案点出后果）

## 常见反例

- 「您输入的内容有误」「您的请求未能成功完成」这种责怪式或翻译腔
- 「确定删除？」按一下就真删了，无撤销路径
- 表单提交失败，错误信息只显示一次1秒消失
- 删除后没有撤销路径，用户后悔无救
- 成功后过于浮夸（「太棒了！您完成了！🎉🎉🎉」） 或过于冷漠（无任何提示）
- 危险操作（永久删除）按钮跟普通按钮长得一样
- 加载失败后没有重试按钮，要用户自己刷新页面
- 收费/扣款没有明显二次确认

## 静态分析能看到的

- 错误文案的字符串内容
- 删除/清空操作前有没有 confirm / modal 弹出
- toast / message 调用的 duration 参数
- 危险按钮是否有 danger / error / warning 类名
- 撤销逻辑（undo、restore、recover）是否存在

## 静态分析看不到的

- 错误信息的真实情绪影响
- 成功反馈的视觉张力

实跑时观察：故意输错看错误信息是否友好；执行删除是否有充分确认；失败后的下一步是否清晰

## 改进方向词

- 错误信息改解释式（参考 ux-writing/error-messages.md）
- 不可逆操作用 modal 二次确认 + 确认文案说清楚后果
- 删除给5-10秒撤销窗口（toast 里带「撤销」按钮）
- 失败状态保持可见，给重试按钮
- 成功反馈适度（toast + 图标即可，工具型产品不要彩带）
- 危险按钮视觉差异化（红色填充、警示icon）
- 收费/扣款类操作必有金额确认 modal
