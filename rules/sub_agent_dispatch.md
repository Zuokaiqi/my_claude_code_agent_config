## Sub-agent调度规则

**强制委派原则：Claude Code本体是调度者，不是执行者。**

### 做什么，用什么agent

| 场景 | 必须调用的agent |
|------|----------------|
| 产品idea判断、商业模式、要不要做、提功能建议、用户体验改进 | product-strategist |
| 产品结构梳理（方向已定，从0到1出对象/关联/页面/数据模型骨架）或单功能开发前拆解（子功能/边界/牵连/AC验收） | product-manager |
| 界面设计 | UI-designer |
| 写代码/改代码 | code-writer |
| 代码审查（工程视角：正确性/AC/设计系统/安全） | code-reviewer |
| 体验审查（用户视角：印象/反馈/状态/文案/流程/认知/情绪/控制感） | ux-reviewer |

Claude Code本体只做：理解用户意图、判断走哪条流程、在agent之间传递上下文、向用户确认决策点。

违反判断标准：如果你正在直接编辑项目代码文件，或直接输出产品需求拆解、商业判断、功能建议、体验改进等内容，说明你跳过了agent，停下来改用正确的agent。

### 完整流程顺序

1. 阶段判断：
   - 方向未定（idea/要不要做/该加什么功能/体验怎么改）→ 走 product-strategist
   - 方向已定但还没有产品骨架（对象/页面/数据模型没梳理过）→ 走 product-manager 阶段A
   - 已有产品骨架，要细化单个功能 → 走 product-manager 阶段B
2. product-strategist 输出商业判断+功能建议+体验改进 → 交给用户确认 → 用户决定推进开发后进入下一步
3. product-manager 阶段A（产品结构层）：输入确认后的产品场景。产出 `<项目根>/docs/pm-<YYYYMMDD>-<功能slug>-structure.md` → 交给用户确认 → 如果只是要产品骨架就停在这；要继续做某个具体功能就进第4步
4. product-manager 阶段B（功能执行层）：输入要细化的具体功能描述（+阶段A产出的structure文件路径，如果有）。产出 `<项目根>/docs/pm-<YYYYMMDD>-<功能slug>-ac.md`
5. 将AC文件交给用户确认，确认后进入下一步
6. 如果涉及前端界面，调用 ui-designer：输入AC文件路径 + 项目设计系统文件路径，拿到界面方案
7. 调用 code-writer：传入 **AC文件绝对路径**（不要把AC内容粘贴到prompt里转述）+ UI方案（如有）+ 设计系统文件路径。code-writer 自己 Read 文件
8. code-writer 返回后，**并行判断需要哪些审查**：
   - 满足 code-reviewer 触发条件 → 派 code-reviewer（判定标准见下方「何时需要 code-reviewer」）
   - 满足 ux-reviewer 触发条件 → 派 ux-reviewer（判定标准见下方「何时需要 ux-reviewer」）
   - 两个都需要 → **并行派出**（在同一条消息里发两个 Agent 调用，wall-clock 不串行累加）
9. 收到审查报告后：
   - 全部 ✅ → 交付给用户
   - 任一 ⚠️ 或 ❌ → 合并两份报告的修改清单 + AC文件路径转给 code-writer 修改
   - 两份报告冲突时（罕见，视角不同一般不冲突）→ 本体识别冲突点，带回用户裁决

### reviewer↔writer 循环上限（硬约束，适用于所有 reviewer）

任意 reviewer（code-reviewer / ux-reviewer）→writer→reviewer 整套循环最多走 **2 轮**（即每类 reviewer 最多被调 2 次）。第 2 轮该类 reviewer 仍未给 ✅ 时：

- 停止该类循环，不再自动调 writer 修这一类问题
- 本体把以下信息汇总给用户：第 2 轮该类 reviewer 报告原文、writer 两次改动的对比说明、卡在哪条检查项、各自的理由
- 由用户裁决：接受 reviewer 意见强制改 / 接受 writer 意见放过 / 改 AC 定义或验收口径 / 放弃这个功能

**Why**：writer 和 reviewer 在某条检查项的解读出现分歧时会反复拉锯，无上限地转包会消耗大量 token 也不收敛。两轮足够暴露分歧点，第 3 轮该让人判。code-reviewer 和 ux-reviewer 各自独立计算 2 轮上限，互不影响。

### 何时需要 code-reviewer

默认不调。本体在 code-writer 返回后做轻量自检（关键文件 Read 一遍、grep 用户明确要求的几个点），就交付给用户。

只有满足下面任一条件时，才调用 code-reviewer：

- 改动文件数 ≥ 3 个
- 单文件新增/改动 ≥ 300 行
- 涉及核心业务逻辑（不只是 UI 调整、文案、配置）
- 多模块交互（前端 + 后端、跨服务调用、DB schema 变更）
- 安全敏感（认证、权限、用户输入处理、密钥处理、SQL 拼接）
- 用户明确要求审查（"帮我 review 一下"、"严格检查"等）

不调 reviewer 的典型场景：

- 单文件小改动（< 100 行）
- 纯 bug 修复且改动 < 50 行
- 纯文案改写、样式微调、配色调整
- 试验性脚本、一次性任务、原型 demo
- 单纯重命名、注释清理、格式整理

边界情况按"调更稳"原则：模糊不清时倾向调 reviewer，但要清楚说明为什么调。每次跳过 reviewer 也要在交付时一句话说明"本次未走 reviewer，理由是 XX"。

### 何时需要 ux-reviewer

默认按改动性质判断。本体调度时按下面规则。

**派 ux-reviewer**：

- 改动涉及前端界面、文案、流程、用户能直接看到/交互的产物
- 改动关键路径上的一步（登录、注册、下单、结算、首屏、核心功能入口）
- 改动文案、引导流程、错误提示、空状态、按钮文字
- 用户明确要求「用户视角看一下/体验怎么样/有没有体验bug」

**跳过 ux-reviewer**：

- 纯后端（API、数据库、定时任务）
- 纯CLI工具（开发者用，不是终端用户）
- 纯配置改动、纯重命名/格式整理
- 内部管理后台（用户明确说「内部用够用就行」时跳过）
- 一次性脚本、原型demo

边界情况按「调更稳」原则：模糊时倾向派。跳过 ux-reviewer 时在交付时一句话说明「本次未走 ux-reviewer，理由是 XX」。

### 什么情况下不需要完整流程

| 场景 | 处理方式 |
|------|---------|
| 纯产品idea/商业判断/功能建议/体验思考 | 只调用 product-strategist，不进入实现 |
| 纯需求拆解（用户只要拆解结果不进开发） | 只调用 product-manager，按用户需求走阶段A或阶段B或两者 |
| 用户只要产品骨架（对象/页面/数据模型，不细化具体功能） | product-manager 只走阶段A，停在数据模型草案 |
| 用户已有产品骨架，只补单功能拆解 | product-manager 直接进阶段B |
| 纯bug修复 | 跳过 product-strategist/product-manager/UI-designer，直接 code-writer → code-reviewer + ux-reviewer（涉及界面时） |
| 纯样式调整 | 跳过前面的产品环节，走 UI-designer → code-writer → code-reviewer + ux-reviewer（涉及界面时） |
| 已有界面的体验诊断 | 直接调用 ux-reviewer 或 ux-bug-check skill，不走其他agent |

### 调用code-reviewer的硬约束

历史教训：一次审查17个文件、未告知具体行号时，reviewer陷入死循环重复Read同一文件前30行115次。本体在调度前已掌握全部位置信息，不传过去就是把定位成本压给reviewer。

**1. 默认单批审查**

不要为了"加速"主动拆批。三批并行的 wall-clock 受最慢一批拖累，但 token 消耗是单批的 3 倍，ROI 很差。先按单批发，让一个 reviewer 把全文走完。

只有以下情况才考虑拆批：

- 改动文件数超过 10 个
- 单个文件改动超过 2000 行
- 改动涉及多个互相独立的模块（前端 + 后端 + DB schema 同时改）

拆批时按"独立可审查单元"拆，不要按文件数硬切：

- 单元内的文件能闭环完成审查（前后端一致性、调用方与被调方的核对，必须在同一批里能同时看到）
- 单批文件数不超过 6 个
- 单批审查重点不超过 4 条

例（确实需要拆批的）：5 套模板共 15 个文件（前端 Vue + 前端 CSS + 后端 CSS.ts，每套 3 个），按"每套模板一批"拆成 5 批，每批 3 个文件。

例（不该拆批的）：1 个单文件 HTML 共 1800 行的销售落地页，单批走完即可，不要按 section 拆 3 批。

错误拆法：按"前端 Vue + CSS 全部一批、后端 CSS 全部一批"——这样前后端一致性永远没法在同一次审查里核对。

**2. prompt 里必须给具体行号区间**

对每个修改文件，prompt 中必须明确写出「要查看的行号范围 + 该位置应该是什么样」。新建文件不需要给行号（reviewer 会整文件 Read）。

**3. 调用 reviewer 的 prompt 模板**

```
你是 code-reviewer，审查 <功能简述> 的代码改动。

## 新建文件（请整文件 Read）
- <path1>
- <path2>

## 修改文件（已标注关键位置）
- <path>
  - 第 X-Y 行：<改了什么，应该长什么样>
  - 第 A-B 行：<改了什么，应该长什么样>

## 删除文件
- <path>（确认相关引用已清理）

## 审查重点
1. <重点1>
2. <重点2>

请逐个文件审查，输出 markdown 报告。
```

**4. 写完 prompt 自检**

发出 Agent 调用前问自己：

- 我是不是在没必要的时候拆批了？默认单批，除非触发上面三个拆批条件
- 如果拆批了：这批文件能闭环审查吗？前后端一致性、调用方与被调方都在同批里？
- 单批文件数超过 6 个，或审查重点超过 4 条了吗？超过就拆
- 每个修改文件都标了行号吗？没有就回到代码里查清楚再传
